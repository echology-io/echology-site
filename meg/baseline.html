Mira Energy Group - MVP Architecture

CTO Technical Strategy Document | Kyle Vines | March 2026

1. Platform Overview

MEG is a software-first platform for decentralized peer-to-peer energy trading. The MVP proves four capabilities in a single integrated system:

  1. Permissioned blockchain settlement - auditable, immutable trade records on a purpose-built energy chain
  2. AI-optimized forecasting - demand/supply prediction enabling smarter trading decisions
  3. IoT data ingestion - real-time smart meter data flowing from edge devices to the platform
  4. Matching engine - automated order matching with transparent price discovery
Regulatory constraint: True P2P energy sales are blocked by utility franchise laws in most US jurisdictions. The MVP targets behind-the-meter scenarios (apartment complexes, campuses, microgrids) where these restrictions don't apply, or operates under a utility sandbox program.

4. Core Components

4.1 Settlement Contract (Solidity / EWC)

The settlement contract is the centerpiece. Everything else feeds into or settles through it. On-chain records are minimal and auditable; heavy data stays off-chain.

On-Chain

  • Trade records (seller, buyer, kWh, price, period)
  • Net settlement balances per participant per period
  • Energy attribute certificates (renewable proof)
  • Meter reading hashes (provenance, not raw data)
  • Market rules (price floor/ceiling, grid fees)

Off-Chain

  • Raw meter data (5-15s intervals, TimescaleDB)
  • Order book / bid-ask matching (latency-sensitive)
  • Weather and forecast data
  • User profiles, KYC
  • Real-time grid state

4.2 Matching Engine

Runs off-chain for performance. Two modes for the MVP:

  • Merit-order auction: Sellers ranked by price (lowest first), buyers ranked by price (highest first). Trades matched until supply meets demand. Settlement price = clearing price.
  • Bilateral matching: Prosumers set preferences (buy local solar first, cheapest first, greenest first). Engine matches based on preference + availability.

Matched trades are batched and committed to the settlement contract at each 15-minute interval.

  • Solidity settlement contract deployed to EWC Volta testnet
  • FastAPI backend: user management, meter data API, settlement orchestration
  • PostgreSQL + TimescaleDB schema: users, meters, readings, trades, settlements
  • MQTT ingest pipeline with simulated meter data (Python generator)
  • Basic matching engine (merit-order auction)
  • Unit tests for contract + API + matching logic

Deliverable: Working backend that accepts meter data, matches trades, and settles on-chain. No frontend yet - API only with test scripts.

Phase 2: Intelligence + Interface Months 2-3

  • XGBoost forecasting model trained on public datasets (Pecan Street, UK CLNR)
  • Forecast API endpoint integrated with matching engine (forecast-informed bidding)
  • Edge gateway prototype: RPi4 + Eastron SDM630 submeter reading real data
  • Next.js prosumer dashboard: energy balance, trade history, forecast visualization
  • WebSocket real-time feed via Redis pub/sub
  • Oracle service: automated meter hash submission to settlement contract

Deliverable: End-to-end demo. Real meter data flows from edge to dashboard, trades match and settle on-chain, forecasts inform the market.

Phase 3: Pilot Prep Months 3-4

  • Identify pilot site (behind-the-meter: apartment complex, campus, co-housing)
  • Edge hardware deployment (10 nodes, ~$1,000-1,500)
  • Utility/regulatory conversations for sandbox participation
  • Security audit: smart contract review, API hardening, edge device auth
  • Investor demo: full platform walkthrough with live data
  • EWC mainnet deployment or Hyperledger Besu private chain (if utility requires)

Deliverable: Fundable MVP. Live pilot site, real data, on-chain settlement, investor deck backed by working product.

7. Regulatory Strategy

Why Behind-the-Meter First

Utility franchise laws prevent direct retail P2P energy sales in most US jurisdictions. FERC Order 2222 enables DER aggregation in wholesale markets but does not authorize retail P2P trading. The fastest path to a real pilot:

  1. Behind-the-meter - multi-tenant buildings, campuses, industrial parks. Utility jurisdiction is limited. Prosumers trade within a shared meter boundary.
  2. Utility sandbox - NY REV, TX deregulated market, CA community microgrid programs offer regulatory sandboxes for innovative energy projects.
  3. Bill credits - structure trades as bill credits (like community solar) rather than direct energy sales. Avoids franchise issues.
  4. Energy certificates - trade RECs/EACs on-chain as a parallel revenue stream that sidesteps energy sales regulation entirely.

10. What Makes This Defensible

  • Auditable settlement on a purpose-built energy chain - not a database with a blockchain label. Every trade is verifiable, immutable, and transparent to regulators.
  • Deterministic-first architecture - AI optimizes but doesn't gatekeep. The market works without the forecast. The forecast makes it work better.
  • Edge-to-chain data integrity - meter readings hashed at the edge, verified on-chain. Tamper-evident from source to settlement.
  • Portable contracts - Solidity on EVM means we're not locked to any single chain. If a utility demands Besu, we deploy to Besu. If regulation favors public chains, we bridge.
  • Regulatory-aware design - behind-the-meter first, bill credits as fallback, certificate trading as parallel revenue. Built to work within constraints, not around them.
