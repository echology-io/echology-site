1. Platform Overview

MEG is a software-first platform for peer-to-peer energy trading. Prosumers (producers + consumers with solar, battery, or both) trade energy within a local market. Every trade settles on a permissioned blockchain with cryptographic proof of the underlying meter data.

Blockchain Settlement

Immutable trade settlement on Energy Web Chain. Every transaction is verifiable by regulators, participants, and auditors.

Anomaly Detection

ML-scored meter readings. Bad data is quarantined before it reaches settlement. The one AI service that's non-negotiable for MVP.

Edge Metering

Signed meter readings from edge devices. Hash chain from physical meter to on-chain settlement. Tamper-evident at every layer.

Matching Engine

Periodic clearing every 15 minutes. Bids and asks collected, clearing price found, trades committed to settlement contract.

Regulatory reality: True P2P energy sales are blocked by utility franchise laws in most US jurisdictions. MEG deploys behind-the-meter first (campuses, multi-tenant, microgrids). No utility permission needed. No regulatory approval needed. Prove the tech, generate real data, expand from there.

Competitive context: Power Ledger has the most commercial traction but uses a token model that creates friction with regulated utilities. LO3/Brooklyn Microgrid proved regulatory constraints make open-grid trading symbolic. SonnenCommunity (acquired by Shell) proved behind-the-meter battery sharing works commercially. No previous platform has combined blockchain settlement with edge-authenticated metering and ML-gated data integrity in a single system.

4. Blockchain Settlement Layer

The settlement layer is the centerpiece. Everything feeds into or settles through it.

4.1 Contract Architecture (4 contracts)

ParticipantRegistry.sol

Role-based access control: Prosumer, GridOperator, Regulator. KYC status tracking. Immutable - no proxy upgrade needed.

EnergyToken.sol (ERC-1155)

Each token ID = generation batch (device + time period). Balance = Wh. Mint triggered by oracle after meter validation. Metadata: source type, location, timestamp.

EnergyMarket.sol

Offer/accept with escrow. Seller locks energy tokens, buyer deposits payment. Atomic settlement releases both sides. Grid fee deduction. Dispute timeout.

Settlement.sol

Periodic settlement (15-min intervals). Validates energy balance (exports ~ imports + losses). Calculates net financial positions. Emits auditable settlement events.

4.2 On-Chain vs Off-Chain

On-Chain (immutable, auditable)

  • Trade records (seller, buyer, kWh, price, period)
  • Net settlement balances per participant
  • Meter reading hashes (provenance)
  • Market rules (price floor/ceiling, grid fees)
  • Participant registry and role assignments

Off-Chain (performant, private)

  • Raw meter data (15-sec intervals)
  • Order book / bid-ask matching
  • User profiles, KYC status
  • Anomaly scores and investigation logs
  • Audit trail (mirrored from on-chain events)

5. Anomaly Detection

The one ML service in the MVP. Bad meter data hitting settlement is the highest-impact risk. The market works without forecasts, without dynamic pricing, without trade recommendations. It does not work if tampered or faulty readings settle as real trades.

5.1 Detection Model

ThreatModelSignals
Meter tamperingIsolation ForestSudden reading drops, statistical deviation from neighbors, zero-injection periods
Equipment failureThreshold + trendFlat readings, communication dropouts, voltage/frequency out of range
Data injectionValidation rulesPhysically impossible values, broken hash chain, invalid signature
Billing fraudStatistical outlierConsumption inconsistent with profile, reverse flow without solar registration

5.2 Response Pipeline

Reading ingested
  -> Validate (physical bounds, signature, timestamp, sequence)
  -> Score (Isolation Forest: anomaly probability 0.0-1.0)
  -> Route:
     score < 0.3  -> Accept, log
     score 0.3-0.7 -> Accept, flag for review, alert operator
     score > 0.7  -> Quarantine, exclude from settlement, investigate
     score > 0.9  -> Quarantine, suspend meter, alert

Runs in-process with the FastAPI app. No separate ML service. Isolation Forest is unsupervised - no labeled training data needed. Fits on pilot data in seconds. Retrain weekly as more data accumulates.

6. IoT / Edge Data Pipeline

6.1 MVP: Simulated Data First

Don't wait for hardware to prove the trading logic. The platform ships with a meter data simulator that generates realistic load profiles (based on Pecan Street public datasets). Simulated readings flow through the same validation, anomaly detection, and settlement pipeline as real readings. When hardware is ready, swap the data source - nothing else changes.

6.2 Edge Node (Phase 2 Hardware)

  • Hardware: Raspberry Pi 4 (4GB) + Eastron SDM630 submeter (~$100/node)
  • Software: Python agent reading Modbus TCP every 15 seconds
  • Local buffer: SQLite ring buffer (24h of readings) for connectivity drops
  • Authentication: Mutual TLS (client certificate per device, issued by MEG CA)
  • Integrity: Each reading signed with device key (ed25519). Hash chain links consecutive readings.
  • Watchdog: systemd watchdog restarts agent on crash. Heartbeat published every 60s.

7. Matching Engine

Periodic Clearing (15-min intervals)

One market mode. No continuous bilateral. No forecast-informed auto-bidding. Keep it simple, auditable, and debuggable.

  1. Trading window opens at interval start
  2. Prosumers submit buy bids (max price, quantity) and sell asks (min price, quantity)
  3. Window closes at interval boundary (aligned to :00, :15, :30, :45)
  4. Engine sorts: sellers ascending by price, buyers descending by price
  5. Clearing price = intersection of supply and demand curves
  6. All matched trades execute at the single clearing price
  7. Matched trades committed to Settlement contract
  8. Unmatched orders expire (no carryover)

Price bounds: Grid operator sets floor and ceiling prices. Floor protects sellers from dumping. Ceiling protects buyers from spikes. Both configurable. Default: floor = $0.04/kWh, ceiling = grid tariff ($0.15/kWh).

8. Settlement Economics

8.1 Payment Model (MVP: Fiat, Off-Chain)

Trades are recorded on-chain. Financial settlement is off-chain via ACH. The blockchain is the ledger of record; the bank is the payment processor.

  • Prosumer links bank account at registration (Plaid)
  • Net settlement calculated per period (15-min intervals, batched daily)
  • Daily net payout/debit via ACH (Stripe Connect or Dwolla)
  • On-chain settlement record serves as the invoice

Lowest regulatory friction - no money transmitter license needed

8.3 Unit Economics

Typical US residential solar prosumer:
  Exported to market:  3,200 kWh/year
  Average trade price: $0.12/kWh (vs $0.15 grid tariff)

Seller:  3,200 x $0.12 = $384 revenue - $96 grid fee - $7.68 platform fee = $280/year net
Buyer:   3,200 x $0.12 = $384 vs $480 grid = $96/year savings

MEG revenue per prosumer pair: ~$15/year
  At 10,000 prosumers:  ~$150K/year
  At 100,000 prosumers: ~$1.5M/year

9. Data Model

Core Schema (PostgreSQL)

CREATE TABLE participants (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    wallet_address  TEXT UNIQUE NOT NULL,
    email           TEXT UNIQUE NOT NULL,
    display_name    TEXT NOT NULL,
    role            TEXT NOT NULL CHECK (role IN ('prosumer','grid_operator','regulator')),
    kyc_status      TEXT DEFAULT 'pending',
    created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE meters (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    participant_id  UUID REFERENCES participants(id),
    device_cert_fp  TEXT NOT NULL,
    model           TEXT,
    source_type     TEXT CHECK (source_type IN ('solar','wind','battery','grid','load')),
    status          TEXT DEFAULT 'active',
    registered_at   TIMESTAMPTZ DEFAULT now(),
    last_heartbeat  TIMESTAMPTZ
);

CREATE TABLE meter_readings (
    id              BIGSERIAL PRIMARY KEY,
    time            TIMESTAMPTZ NOT NULL,
    meter_id        UUID NOT NULL REFERENCES meters(id),
    energy_kwh      NUMERIC NOT NULL,
    net_energy_wh   INTEGER,
    power_kw        NUMERIC,
    voltage_v       NUMERIC,
    sequence_num    BIGINT NOT NULL,
    signature       TEXT NOT NULL,
    reading_hash    TEXT NOT NULL,
    prev_hash       TEXT,
    anomaly_score   NUMERIC DEFAULT 0,
    validated       BOOLEAN DEFAULT false
);
CREATE INDEX idx_readings_meter_time ON meter_readings(meter_id, time);

CREATE TABLE orders (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    participant_id  UUID REFERENCES participants(id),
    side            TEXT NOT NULL CHECK (side IN ('buy','sell')),
    energy_wh       INTEGER NOT NULL,
    price_per_wh    NUMERIC NOT NULL,
    period_start    TIMESTAMPTZ NOT NULL,
    status          TEXT DEFAULT 'open' CHECK (status IN ('open','matched','partial','expired')),
    created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE trades (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    buyer_id        UUID REFERENCES participants(id),
    seller_id       UUID REFERENCES participants(id),
    energy_wh       INTEGER NOT NULL,
    price_per_wh    NUMERIC NOT NULL,
    total_price     NUMERIC NOT NULL,
    grid_fee        NUMERIC NOT NULL,
    platform_fee    NUMERIC NOT NULL,
    period_start    TIMESTAMPTZ NOT NULL,
    chain_tx_hash   TEXT,
    status          TEXT DEFAULT 'pending' CHECK (status IN ('pending','settled','disputed')),
    settled_at      TIMESTAMPTZ
);

CREATE TABLE settlements (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    period_start    TIMESTAMPTZ NOT NULL,
    period_end      TIMESTAMPTZ NOT NULL,
    total_traded_wh INTEGER NOT NULL,
    clearing_price  NUMERIC,
    total_trades    INTEGER NOT NULL,
    total_grid_fees NUMERIC NOT NULL,
    total_platform_fees NUMERIC NOT NULL,
    chain_tx_hash   TEXT NOT NULL,
    meter_hash_root TEXT NOT NULL,
    settled_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE audit_events (
    id              BIGSERIAL PRIMARY KEY,
    event_type      TEXT NOT NULL,
    actor_id        UUID,
    target_type     TEXT,
    target_id       TEXT,
    payload         JSONB NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_audit_type ON audit_events(event_type, created_at);

6 tables. No hypertables, no continuous aggregates, no materialized views. Standard Postgres handles the pilot. Add TimescaleDB when meter data volume justifies it (Phase 4+).

Privacy (MVP): Meter-level data is PII in some states. Encrypted at rest (Postgres column-level). On-chain records use wallet addresses only - no PII on-chain. Off-chain PII mapping deleted on account deletion. Full privacy architecture is Phase 4.

12. Regulatory Strategy

Expansion Pathway

Phase 1: Behind-the-Meter

No utility franchise issues. Prosumers trade within a shared meter boundary (campus, apartment, microgrid). Platform operates as building energy management.

No regulatory approval needed

Phase 2: Utility Sandbox

Partner with utility under regulatory sandbox or pilot program. Trades structured as bill credits. Utility provides meter data access.

Requires utility partnership

Phase 3: Licensed Retailer

In deregulated markets (TX), become a Retail Electric Provider. In regulated markets, operate as Community Choice Aggregator.

Licensing + capital requirements

Phase 4: Wholesale Market

Aggregate DERs under FERC Order 2222 for ISO/RTO markets (capacity, energy, ancillary services). 100 kW minimum aggregation.

ISO registration + telemetry

Audit Trail

Every action that affects settlement produces an immutable audit record. Logged to PostgreSQL, hashed and anchored on-chain. Regulator gets read access to the operator API.

  • Meter events: reading received, validated, flagged, quarantined
  • Trade events: bid placed, matched, settled, disputed
  • System events: market paused, meter suspended, price bounds changed
  • Retention: 7 years minimum (FERC record retention)

13. Build Plan, Risks & Cost

Phase 1: Core Settlement Months 1-2

  • Solidity contracts on EWC Volta: ParticipantRegistry, EnergyToken (ERC-1155), EnergyMarket, Settlement
  • FastAPI backend: auth, meter data ingestion, order placement, settlement orchestration, audit logger
  • PostgreSQL schema (6 tables above)
  • Simulated meter data generator (realistic load profiles)
  • Periodic clearing engine (15-min intervals)
  • Anomaly detection pipeline (Isolation Forest + validation rules)
  • Contract tests (Hardhat + Foundry, 100% branch coverage, fuzz)
  • API tests (pytest, unit + integration)
  • Deploy to Railway

Exit criteria: 10 simulated prosumers trading. Settlement math correct. Audit trail complete. API-only, no frontend.

Phase 2: Dashboard + Hardware Months 2-3

  • Next.js prosumer dashboard: readings, orders, trades, balance, market price
  • Operator dashboard: market overview, anomaly alerts, meter management
  • WebSocket real-time feed
  • Edge gateway prototype: RPi4 + SDM630, signed readings, local buffer
  • MQTT broker with mTLS
  • End-to-end test: edge to dashboard to chain

Exit criteria: Full demo with real hardware. One RPi feeding live data through the entire pipeline to on-chain settlement.

Phase 3: Pilot Months 3-4

  • Secure behind-the-meter pilot site
  • Deploy 5-10 edge nodes
  • Fiat settlement integration (Stripe Connect, ACH)
  • Smart contract security audit (external firm)
  • Regulatory documentation package
  • Investor demo with live data

Exit criteria: Real prosumers, real data, real money, security audited. Ready for pre-seed raise.
